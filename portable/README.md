# Megaton Hammer portable dialogue (open, fork-independent) — MIT

This folder is **standalone reference source** you can drop into *any* OoT-based project — vanilla
decompilation, Ship of Harkinian, 2Ship, HackerOoT, or a total-conversion mod. It is **not** tied to
Megaton Hammer's own SoH/2Ship forks. The editor authors the dialogue; this is how the *behaviour* runs.

## The split

- **Presentation is universal, zero code.** Message text, colour, timing, inline SFX, two-choice prompts,
  and branch-to-message (`CTRL_TEXTID`) are all standard game message control bytes. Megaton Hammer emits
  them into the normal message data (in-place ROM append for carts; the runtime message resource for
  SoH/2Ship). Signs, POIs, readable objects, and branching NPCs need **nothing** from this folder.

- **Outcomes need code**, because "give an item / charge rupees / set a flag / branch on a choice" is NPC
  behaviour the message format can't carry. That behaviour is expressed as a small **data table**
  (`mh_dialogue.h`) + one generic **actor** (`ovl_En_MhTalk`). You compile both into your base. Where a real
  vanilla actor already does the job (a shop = browse+charge+give), just reskin that instead.

## Files

| File | What it is |
|---|---|
| `mh_dialogue.h` | The `MhDialogueEntry` table format + `MhDialogue_Find`. The stable contract. |
| `ovl_En_MhTalk/z_en_mhtalk.c` | A generic invisible "talk point": on talk it shows the message and applies the outcome, with the fulfilled-state fallback. |
| `mh_dialogue_data.c` | **Generated by Megaton Hammer** per project — the `gMhDialogueTable[]` rows. (Editor export; not in this folder.) |

## Integrate

1. Register the overlay as **`ACTOR_EN_MHTALK` at actor id `0x0230`** (the id Megaton Hammer places it at —
   `ActorDatabase.MhTalkId`) and add an overlay-table row for it. Using a different id? Change it in the
   editor's actor DB to match.
2. Add `z_en_mhtalk.c` and the editor-generated `mh_dialogue_data.c` to your build (and `mh_dialogue.h` to your include path).
3. The editor places it with **params = a dialogue slot (0–255)**; the entry message textId is
   **`0x1000 + slot`** (`MH_DIALOGUE_TEXTID_BASE`). The message text for that textId comes from your base's
   message system (the runtime CustomMessage resource on SoH/2Ship, or your decomp message table) — that
   textId range (`0x1000+`) is custom, so it won't collide with vanilla text.
4. Author the message text (colour/timing/SFX/prompt) + outcomes in the Dialogue Editor; export writes both
   the message bytes and `mh_dialogue_data.c`.

The actor uses only stock decomp API (`Actor_OfferTalk`, `Message_StartTextbox`, `Message_GetState`,
`Flags_Get/SetSwitch`, `Rupees_ChangeBy`, `Actor_OfferGetItem`). A few names differ slightly across bases
(e.g. `Actor_OfferGetItem` vs `func_8002F434`) — adapt to your tree's headers.

## Data format

```c
typedef struct {
    u16 textId;         // the message this row describes
    u16 sfx;            // 0 = none
    s8  gesture;        // -1 = default; else a supporting NPC's sAnimationInfo[] index (see docs/dialogue-npc-gestures.md)
    u8  isPrompt;       // 0 = message (outcome[0] on advance), 1 = two-choice
    MhOutcome outcome[2]; // [0] advance / "Yes", [1] "No"; each: {nextMsgId, fireFlag, giveItem, rupeeCost} (-1/0 = none)
    s16 doneFlag;       // when set, show afterMsgId instead of textId
    s16 afterMsgId;
} MhDialogueEntry;
```

### Example — Talon's "buy a Cuccoo for 20 Rupees"

```c
const MhDialogueEntry gMhDialogueTable[] = {
    /* greeting: plain message, plays a laugh SFX, then advances to the prompt */
    { 0x0301, 0x6836, /*gesture*/ 1, /*isPrompt*/ 0,
      { { 0x0302, -1, -1, 0 }, { -1, -1, -1, 0 } }, /*done*/ -1, -1 },

    /* prompt: Yes = give Pocket Cuccoo + charge 20 + set done flag 12; No = goto 0x0303 */
    { 0x0302, 0x0000, -1, /*isPrompt*/ 1,
      { { -1, /*flag*/ 12, /*item*/ GI_POCKET_CUCCO, /*rupees*/ 20 },
        { 0x0303, -1, -1, 0 } }, /*doneFlag*/ 12, /*afterMsgId*/ 0x0304 },
};
const s32 gMhDialogueCount = ARRAY_COUNT(gMhDialogueTable);
```

Once flag 12 is set (the purchase completed), the talk point shows `0x0304` ("Thanks! Take good care of
that Cuccoo.") instead of the offer — the classic "already fulfilled" line.

## Gestures

Gestures are **actor-internal** in the games (each NPC picks animations by its own textId/state — see
`docs/dialogue-npc-gestures.md`). `En_MhTalk` is a modelless talk point, so its `gesture` field is a hint
for a *supporting* NPC or a future model-carrying variant, not something it animates itself. Reskinning an
existing NPC keeps that NPC's own gestures automatically.
