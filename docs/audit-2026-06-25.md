# Megaton Hammer — stub / unfinished-feature audit (2026-06-25)

Sweep for stubs, dead UI, deferred functionality, and features that don't fully meet their intended
scope. Three dimensions: UI controls, non-UI logic/export/render, and SDK2013-Hammer tool parity.

## Summary

The codebase is in good shape. **No dead or no-op UI controls were found** — every menu item,
toolbar button, and dialog button has a wired, functional handler; the few empty handlers are
intentional (documented no-ops, e.g. single-click-doesn't-switch-room, preset-combo loads via an
explicit button). The only behavioural correctness gap found is now **fixed** (alt-header 0x0B,
below). Everything else is either intentional design or unbuilt larger features (tracked in
`hammer-parity-audit.md`).

## Fixed this pass

- **Room export: missing 0x0B object-dependency list on the alternate-header path** (`RoomExporter.
  BuildWithSetups`). Rooms with alternate setups (time-of-day / event variants) emitted no per-header
  object list, so an actor unique to one setup could fail to spawn (its object never loaded). Now each
  header (primary + alternates) emits its own 0x0B array; header sizes/offsets computed per-header.
  Backward-compatible (no objResolver → no 0x0B, byte-identical to before). Regression-checked via
  soh_harness (RENDER OK, no crash).

## Intentional by design (not gaps — documented rationale)

- **Scene export writes placeholder room VROMs (zeros)** — patched by `RomInjector.PatchRoomList`;
  the export format is an injector input, not a standalone ROM patch.
- **Cutscene / NPC-schedule / surface-camera / per-water-light data is retain-only or default** — these
  are preserved on round-trip but have no from-scratch authoring UI. Schedules ARE authorable for the
  custom engine (O2R `mh/schedules`); ROM-injection schedules are out of scope by platform.
- **Rendering approximations (editor viewport only, never affects export):** lit-model shading
  approximates the RDP combiner; CI/palette textures without a loaded TLUT fall back to grayscale;
  unmodeled actors draw item-icon billboards. All cosmetic, all acknowledged in-code.
- **Cross-game audio curation** — a few soundfont-incompatible songs are excluded from the cross-game
  list deliberately.

## SDK2013-Hammer parity

The per-tool parity audit (`hammer-parity-audit.md`) was reconciled: its body was the original
pre-implementation audit and is stale on items since completed (marquee, groups, visgroups, camera
gizmos, block shapes, Face Edit, vertex multi-select, path polyline). The genuinely-unimplemented set
is now listed accurately there — all are unbuilt features or cosmetic polish (CSG carve/hollow, brush
entities [N/A for OoT], 3D brush manipulation, clip keep-side shading, cordon, pane-maximize), none a
broken control.

## Freestanding items (vanilla-compatibility follow-up)

Truly *arbitrary* freestanding GetItems are **not vanilla-possible** (only the SoH/2Ship rando fork's
`ITEM00_SOH_GIVE_ITEM_ENTRY` + GetItemEntry override does that) — so per the vanilla-only constraint it
was NOT implemented. Vanilla coverage is instead: chests give **any** GetItem; `En_Item00` covers its
spinning-model set; and **`Item_Etcetera` (0x010F) was added** this pass as a configurable freestanding
actor (bottle, Ruto's letter, Hylian shield, quiver-40, silver/gold scale, small key, fire arrow).
